## 🛠️ รายการฟังก์ชันใหม่ที่พัฒนาเพิ่มเติม (New Features Added)

### 1. ระบบผู้ดูแลระบบย่อยของส่วนกลาง (System Sub-Admin)
* **บทบาทสิทธิ์ (Role Definition)**: เพิ่มบทบาท `SYSTEM_SUB_ADMIN` (System Sub-Administrator) ในโครงสร้างของระบบ
* **สิทธิ์การทำงาน (Permissions)**:
  * **ต่ำกว่า System Admin**: มีสิทธิ์ดูแลจัดการระบบโดยทั่วไปได้ทั้งหมด (เช่น ดูรายการตั๋วและแก้ไขของทุกบริษัท, ดูรายชื่อสมาชิกของทุกบริษัท, จัดการรายชื่อบริษัท/องค์กร)
  * **ข้อจำกัดสิทธิ์**:
    * **ไม่สามารถ** เข้าถึงหน้าตั้งค่าระบบ SMTP (`/settings/`) ได้ (ต้องเป็น `SYSTEM_ADMIN` หรือ `Superuser` เท่านั้น)
    * **ไม่สามารถ** สร้าง แก้ไข หรือลบผู้ใช้งานที่เป็นระดับ `SYSTEM_ADMIN` หรือ `SYSTEM_SUB_ADMIN` คนอื่นๆ ได้ (จำกัดสิทธิ์ให้สร้างได้เฉพาะ `CLIENT_ADMIN` และ `CLIENT_USER` เท่านั้น)
    * **การป้องกัน**: ควบคุมความปลอดภัยตั้งแต่ระดับฟอร์มกรอกข้อมูล (`CustomUserForm`) และที่ระดับ `get_object()` / `form_valid()` บน View เพื่อป้องกันการโจมตีข้ามระดับสิทธิ์ผ่านการยิง API/Post โดยตรง

### 2. ระบบตั้งค่าการส่งอีเมล SMTP ผ่านหน้าเว็บ (Dynamic SMTP Settings Page)
* **ตารางฐานข้อมูลใหม่**: เพิ่มตาราง `SMTPConfiguration` สำหรับบันทึกรายละเอียดของเซิร์ฟเวอร์ผู้ให้บริการส่งเมลในอดีตและปัจจุบัน
* **ฟีเจอร์หลักบนส่วนติดต่อผู้ใช้งาน**:
  * **Preset บริการด่วน (Gmail / Microsoft Presets)**: เมื่อคลิกเลือกประเภทผู้ให้บริการ ระบบจะมี JavaScript ช่วยกรอกข้อมูลอัตโนมัติ:
    * **Gmail**: Host=`smtp.gmail.com`, Port=`587`, TLS=`True`
    * **Microsoft Outlook**: Host=`smtp.office365.com`, Port=`587`, TLS=`True`
    * **Simulation**: Host=`localhost`, Port=`1025`, TLS=`False`
    * **Custom**: ผู้ใช้ป้อนข้อมูลเซิร์ฟเวอร์อื่นๆ เองได้ตามต้องการ
  * **ระบบล็อกสิทธิ์การใช้งาน (Single-Active Config)**: ระบบบังคับใช้เงื่อนไขให้สามารถเปิดใช้งานเซิร์ฟเวอร์ SMTP ได้ **สูงสุดเพียง 1 ตัวพร้อมกันเท่านั้น** (หากผู้ใช้กดเปิดเซิร์ฟเวอร์ใหม่ ตัวเก่าที่ทำงานอยู่จะถูกตั้งเป็น Inactive อัตโนมัติที่ระดับฐานข้อมูลผ่าน `save()`)
  * **การจัดการครบวงจร (CRUD)**: สร้างหน้าจัดการ `/settings/` เพื่อเพิ่มการตั้งค่า, สลับปิด/เปิดใช้งานชั่วคราว, กดปุ่มแก้ไขดึงข้อมูลย้อนกลับมาบนฟอร์ม, หรือกดลบทิ้งได้จากหน้าจอระบบทันที

### 3. ระบบ Dynamic Email Dispatcher Backend
* **การทำงานเบื้องหลัง**:
  * เมื่อระบบต้องส่งอีเมล (เช่น มีการตั้งตั๋วใหม่, เปลี่ยนแปลงสถานะงาน หรือจัดส่งรายงาน PDF ประจำเดือนแบบด่วน) ระบบจะทำการค้นหาการตั้งค่า SMTP ที่เป็น **Active** ในระบบก่อน
  * หากพบเซิร์ฟเวอร์ที่เป็น Active และไม่ใช่ SIMULATION: ระบบจะสร้างอินสแตนซ์ `EmailBackend` ของ Django ด้วยค่าเซิร์ฟเวอร์นั้น (host, port, username, password, use_tls) เพื่อส่งเมลจริงออกไป
  * หากไม่มีเซิร์ฟเวอร์ที่เปิดใช้งานอยู่: ระบบจะตกลงกลับไปใช้งาน backend ตั้งต้นที่ระบุใน `.env` หรือ `settings.py` (เช่น simulation บน console) โดยอัตโนมัติเพื่อความเสถียรสูงสุด
  * **สามารถเลือกอีเมลส่งได้**: ตัวระบบกำหนดให้ `from_email` ปลายทางที่ใช้ส่งออก จะเปลี่ยนตามอีเมลผู้ใช้งาน (Username) ของเซิร์ฟเวอร์ SMTP ตัวที่เปิดใช้อยู่โดยตรง ทำให้หมดปัญหาการถูกตีกลับ (Dynamic From Header)

### 4. ระบบการเลือกอีเมลผู้ส่งตามความต้องการ (Send From Account Selector)
* **กล่องตัวเลือก (Dropdown Selector)**: เพิ่มส่วนเลือกอีเมลผู้ส่ง SMTP บนหน้าจอ Dashboard ประจำเดือน เมื่อกดส่งรายงานผู้ดูแลระบบสามารถเจาะจงเลือกบัญชีผู้จัดส่ง SMTP ได้ หรือเลือกใช้เซิร์ฟเวอร์หลัก (Active System Configuration)
* **การเชื่อมต่อและดึงสิทธิ์แบบ Dynamic**: ระบบจะดึงและสลับ `EmailBackend` และ `from_email` ไดนามิกไปตามเซิร์ฟเวอร์ SMTP ตัวที่ผู้ใช้กำหนดบนฟอร์มทันที

### 5. ระบบเพิ่มหมวดหมู่ของ Ticket (Ticket Category Field)
* **หมวดหมู่ปัญหา (Categories)**: พัฒนาเพิ่มฟิลด์ `category` ในตาราง `Ticket` โดยจัดแบ่งออกเป็น 5 หมวดหมู่หลัก ได้แก่:
  1. **Hardware** (อุปกรณ์คอมพิวเตอร์ / เครื่องคอมพิวเตอร์)
  2. **Software** (โปรแกรม / ระบบซอฟต์แวร์)
  3. **Network / Internet** (ระบบเครือข่าย / เครือข่ายอินเทอร์เน็ต)
  4. **Account / Login** (บัญชีผู้ใช้งาน / การเข้าระบบ)
  5. **Other** (อื่นๆ)
* **การนำเสนอและจัดเก็บข้อมูล**: ฟิลด์ถูกเพิ่มเข้าไปในฟอร์มการสร้างตั๋ว การแก้ไขตั๋ว การ์ดรายละเอียด และถูกแสดงเป็นคอลัมน์ใหม่บนหน้า Dashboard หลัก

### 6. ระบบการสลับเปลี่ยนภาษา TH / EN (Bilingual Language Switcher)
* **การสลับภาษาแบบเรียลไทม์**: พัฒนา Custom Context Processor ดึงการตั้งค่าตัวเลือกภาษาผ่าน Cookie `lang` (เริ่มต้นใช้งานคือ `th`)
* **สวิตช์ TH / EN**: ติดตั้งปุ่มสวิตช์ภาษาบริเวณขวาบนของ Header ระบบ ในทุกหน้าที่ผู้ใช้ล็อกอินใช้งาน
* **การแปลเนื้อหา**: ทำการปรับปรุงข้อความส่วนกลางของระบบ (สไลด์บาร์นำทาง, แผงควบคุมสถิติ, หัวข้อคอลัมน์, ฟอร์มป้อนข้อมูล, ป้ายกำกับฟิลด์, ปุ่มยืนยัน) ให้เป็นไดนามิกสลับภาษาได้ทันทีตามค่า Cookie ที่ถูกเก็บ

### 7. ระบบเข้าใช้งานด้วย Username หรือ Email และปุ่มเปิด/ปิดตาดูรหัสผ่าน
* **การล็อกอินยืดหยุ่น**: ระบบรองรับการเข้าสู่ระบบผ่านการป้อนข้อมูลได้ทั้ง **ชื่อผู้ใช้งาน (Username)** หรือ **อีเมล (Email)** โดยใช้ Custom Authentication Backend ค้นหา Case-insensitive
* **ปุ่มสลับการมองเห็นรหัสผ่าน (Password Visibility Toggle)**: ติดตั้งปุ่มตา (Eye toggle button) สำหรับสลับซ่อน/แสดงรหัสผ่านในทุกจุดกรอกรหัสผ่าน (หน้า Login, หน้าสร้าง/แก้ไขผู้ใช้, และหน้าจัดการตั้งค่า SMTP)
* **การจัดวาง Layout บนมือถือ**: ปรับปรุงลำดับความสูง (Stacking Context) บน `base.html` โดยเพิ่มคลาส `relative z-20` ให้แถบ Header ป้องกันส่วนโหมดแสดงผลธีมและสีถูกการ์ดหรือเนื้อหาอื่นๆ ทับบนหน้าจอมือถือ

### 8. ระบบไฟล์แนบสำหรับแต่ละตั๋ว (Ticket File Attachment)
* **การบันทึกไฟล์แนบและการจำกัดขนาด (10 MB Limit)**: เพิ่มฟิลด์ `attachment` ลงใน `Ticket` รองรับการเลือกแนบไฟล์ประกอบความกว้างไม่เกิน 10 MB โดยระบบจะทำการตรวจเช็คขนาดไฟล์ในฝั่งเซิร์ฟเวอร์แบบสองภาษา (Bilingual validation) และแจ้งเตือนหากไฟล์มีขนาดใหญ่กว่าที่กำหนด ไฟล์ที่บันทึกสำเร็จจะเก็บไว้ที่ `/media/ticket_attachments/`
* **การเสิร์ฟไฟล์**: ปรับแต่ง `settings.py` และ `urls.py` เพื่อเสิร์ฟไฟล์สื่อสิ่งพิมพ์ต่างๆ (Media files) ผ่านพาร์ท `/media/` ได้ในระดับโลคอล
* **ดาวน์โหลด**: แสดงลิงก์และปุ่มดาวน์โหลดไฟล์แนบพร้อมไอคอนคลิปหนีบกระดาษที่หน้ารายละเอียดตั๋วโดยอัตโนมัติหากตั๋วมีไฟล์แนบ

### 9. ระบบกล่องความคิดเห็นและการแจ้งเตือนผ่านอีเมล (Comments & Email Notification)
* **การพูดคุยในตั๋ว**: เพิ่มกล่องแสดงความคิดเห็น (Comment Timeline) ที่หน้ารายละเอียดตั๋ว แสดงรายการข้อมูลประวัติการตอบกลับเรียงตามลำดับเวลา
* **ส่งการแจ้งเตือนทางอีเมล**: เมื่อมีการพิมพ์ข้อความตอบกลับใหม่เกิดขึ้น ระบบจะส่งอีเมลแจ้งเตือนผู้ที่เกี่ยวข้องโดยอัตโนมัติ:
  * **ผู้แจ้งปัญหา (Creator)**: ได้รับอีเมลแจ้งเตือน (กรณีที่ผู้พิมพ์ข้อความตอบกลับไม่ใช่ผู้แจ้งปัญหาเอง)
  * **ช่างเทคนิคผู้ดูแล (Assignee)**: ได้รับอีเมลแจ้งเตือน (กรณีที่ผู้พิมพ์ข้อความไม่ใช่ช่างเทคนิคคนนั้น และตั๋วได้รับการมอบหมายแล้ว)
* **การทำงานร่วมกับ Dynamic SMTP**: ระบบจะใช้การเชื่อมต่อเมล SMTP ของผู้ส่งตามบัญชีเซิร์ฟเวอร์หลักที่กำลัง **Active** ในขณะนั้น หรือตามที่ระบบถูกสั่งการ และเก็บบันทึกประวัติความสำเร็จไว้ใน `EmailLog` (ระบุประเภทเป็น `COMMENT_ADDED`)

## 🛠️ โครงสร้างไฟล์การติดตั้งจริง (Deployment Scripts)

เราได้สร้างโฟลเดอร์สำหรับนำแอปขึ้นเซิร์ฟเวอร์โปรดักชันจริงไว้ใน [deployment/](file:///d:/Project_personal/ticketSolve/deployment/) เพื่อช่วยอำนวยความสะดวกในการดีพลอยแอปดังนี้:
* `gunicorn.service`: ไฟล์ systemd service daemon สำหรับควบคุมแอปในเบื้องหลัง
* `nginx.conf`: บล็อกการตั้งค่าเซิร์ฟเวอร์บล็อก Nginx คอยจัดการ Reverse Proxy และไฟล์ Static/Media
* `deploy.sh`: สคริปต์อัตโนมัติรันบน VPS Ubuntu เพื่อสั่งติดตั้ง Virtual Environment, library ต่างๆ และย้ายค่า config ให้อัตโนมัติ
